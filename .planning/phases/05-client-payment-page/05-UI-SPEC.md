# Phase 5 — UI Design Contract: Client Payment Page

> Visual and interaction contract for the public /pay/{uuid} experience.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> This is the client-facing payment page — the "life of this project."
> Design standard: Stripe Checkout / Linear / Vercel level of polish.
> NOT a functional stub. Every pixel, state, and word is specified.

---

## Scope

Four Vue pages, all using `PaymentLayout.vue` (new, standalone, no auth):

| Page | Route | File |
|------|-------|------|
| Payment form | `GET /pay/{uuid}` | `resources/js/pages/ClientPayment/Pay.vue` |
| Success | `GET /pay/{uuid}/success` | `resources/js/pages/ClientPayment/Success.vue` |
| Failure | `GET /pay/{uuid}/failed` | `resources/js/pages/ClientPayment/Failed.vue` |
| Guard (non-pending) | rendered by controller | `resources/js/pages/ClientPayment/Unavailable.vue` |

Layout file: `resources/js/layouts/PaymentLayout.vue`

This design system is SEPARATE from the admin panel. Do not extend `AppLayout.vue` or any auth layout.

### Brand Theming System (pre-decided — D-08)

Brand colors are injected as CSS custom properties on the layout root element:

```html
<div
  :data-brand="brand.slug"
  :style="`--brand-primary: ${brand.primary_color}; --brand-secondary: ${brand.secondary_color}`"
>
```

The `brand-theme.css` `[data-brand]` rule forwards these to `--color-brand-primary` and `--color-brand-secondary`.

The Stripe Elements appearance object references `--brand-primary` directly (see Stripe Elements section below).

- Background: `bg-muted/40` — very subtle off-white, keeps focus on the card
- Card max-width: `max-w-md` (448px) — matches AuthCardLayout established precedent
- Footer: "Secured by Stripe" with `LockIcon` from lucide-vue-next, `text-xs text-muted-foreground`
- No dark mode toggle on public page — always light mode (`class="light"` on html root or handle via body)

## Spacing Scale

Tailwind CSS 4 spacing — all multiples of 4px:

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `gap-1` / `p-1` | Icon gaps, badge padding |
| sm | 8px | `gap-2` / `p-2` | Inline element spacing |
| md | 16px | `gap-4` / `p-4` | Default element spacing, form field gaps |
| lg | 24px | `gap-6` / `p-6` | Card internal padding (CardContent default is `px-6`) |
| xl | 32px | `gap-8` / `p-8` | Logo-to-card gap |
| 2xl | 48px | `gap-12` | Major section breaks |
| 3xl | 64px | `gap-16` | Page-level vertical centering padding |

**Exceptions:**
- Logo image height: `h-10` (40px) — touch-target exception
- "Secured by Stripe" footer top margin: `mt-8` (32px)
- Card internal form gap: `space-y-4` (16px between fields), `space-y-6` (24px between major sections)
- Stripe Elements `#payment-element` container: `mt-4 mb-6` (16px top, 24px bottom)

### Stripe Elements Appearance Object

```typescript
const appearance = {
  theme: 'stripe',
  variables: {
    colorPrimary: getComputedStyle(document.documentElement)
      .getPropertyValue('--brand-primary').trim()
      || '#000000',
    colorBackground: '#ffffff',
    colorText: 'hsl(0, 0%, 3.9%)',
    colorDanger: 'hsl(0, 84.2%, 60.2%)',
    fontFamily: '"Instrument Sans", ui-sans-serif, system-ui, sans-serif',
    borderRadius: '6px',   // matches --radius-md (calc(0.5rem - 2px))
    spacingUnit: '4px',
  },
  rules: {
    '.Input': {
      border: '1px solid hsl(0, 0%, 92.8%)',   // --border
      boxShadow: 'none',
    },
    '.Input:focus': {
      border: '1px solid var(--brand-primary)',
      boxShadow: '0 0 0 3px color-mix(in srgb, var(--brand-primary) 20%, transparent)',
    },
    '.Label': {
      fontSize: '14px',
      fontWeight: '400',
      color: 'hsl(0, 0%, 3.9%)',
    },
    '.Error': {
      fontSize: '14px',
    },
  },
}
```

**Note:** `colorPrimary` must be read at mount time via `getComputedStyle` — not hardcoded — because each payment page has a different brand's primary color injected as a CSS variable.

## States and Interactions

### Pay.vue State Machine

| State | Button text | Button state | Form state | Notes |
|-------|-------------|--------------|------------|-------|
| `idle` | "Pay $X.XX" | enabled | interactive | Elements mounted, ready |
| `processing` | "Processing..." | disabled + spinner | pointer-events-none | After `stripe.confirmPayment()` called |
| `error` | "Pay $X.XX" | re-enabled | interactive | Show Alert with error message |
| `redirect` | (leaving page) | — | — | Stripe redirects to success/failed route |

### Elements Loading State

Before Stripe Elements mounts:

```html
<div class="h-32 flex items-center justify-center">
  <Spinner class="size-5 text-muted-foreground" />
</div>
```

Replace with `<div id="payment-element" />` after `elements.create()` completes. Use `onReady` event from Stripe Elements to swap the skeleton.

### Error Alert

Use shadcn-vue `Alert` component (already available):

```html
<Alert v-if="errorMessage" variant="destructive" class="mt-2">
  <AlertCircle class="size-4" />
  <AlertDescription>{{ errorMessage }}</AlertDescription>
</Alert>
```

`errorMessage` is set from `confirmPayment()` result: `result.error?.message ?? 'An unexpected error occurred. Please try again.'`

## Mobile Layout (CLIENT-08)

All four pages must render correctly on screens ≥ 320px wide.

| Breakpoint | Layout behavior |
|------------|-----------------|
| `< 640px` (mobile) | Full-width card, `px-4` page padding, logo `h-8`, card has `rounded-xl` still |
| `≥ 640px` (sm+) | Centered `max-w-md`, `px-4` page padding |
| `≥ 768px` (md+) | Same as sm — this is a single-column layout permanently |

**Mobile-specific rules:**
- Submit button: `size="lg"` (`h-10`) — tall enough for thumb tap (meets 44px touch target minimum at 40px + padding)
- "Try again" button: same `size="lg"`, `w-full`
- Stripe Elements: single column automatically — `layout: 'auto'` in Elements options handles this
- Text does not scale below `text-sm` on mobile — no `text-xs` for interactive labels
- Logo capped at `h-10 max-w-[180px]` to prevent oversized logos breaking layout on small screens

## Security Constraints (visual implications)

These security rules from CLAUDE.md have direct UI consequences:

| Rule | UI Implication |
|------|---------------|
| `clientSecret` never in URL | Stripe `return_url` points to `/pay/{uuid}/success` — success controller reads `?redirect_status` param to confirm. The `client_secret` query param Stripe appends is read server-side then discarded — never re-rendered. |
| Amount always server-side | Pay.vue shows amount from Inertia props. No editable amount field exists anywhere on this page. |
| No DB writes from confirmPayment | Pay.vue never calls any Inertia POST on success — it only follows Stripe's redirect. All status writes are Phase 6 webhooks. |
