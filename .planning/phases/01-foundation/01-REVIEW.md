# Phase 1: Code Review Report

**Reviewed:** 2026-05-03T00:00:00Z
**Depth:** standard
**Files Reviewed:** 22
**Status:** issues_found

## Summary

This review covers the Phase 1 Foundation deliverables: Eloquent models, migrations, factories, seeders, Pest tests, CSS theme scaffolding, and shadcn-vue UI primitives.

The encryption architecture (encrypted cast on `secret_key` / `webhook_secret`, TEXT column type in migration) is correctly implemented and matches the CLAUDE.md security requirements. The `amount` integer cast is correct. The RBAC wiring (spatie/laravel-permission, Fortify, `HasRoles` trait) is properly scaffolded.

Three critical issues require fixes before Phase 2 begins: the `Payment` model does not auto-generate its own UUID, the `PaymentFactory` creates mismatched `brand_id` / `stripe_account_id` pairs, and encrypted credential fields are unnecessarily exposed via mass assignment. Four warnings cover missing route-key hardening, an incomplete test assertion, a weak seeder password, and a potential CSS injection vector in the brand-theme system.

## Critical Issues

### CR-01: Payment UUID Not Auto-Generated by Model

**File:** `app/Models/Payment.php:1-42`

**Issue:** The `uuid` column is populated by `PaymentFactory` but there is no `boot()` / `creating` observer on the `Payment` model itself. Any code path that creates a `Payment` without going through the factory (e.g., a controller doing `Payment::create([...])` in future phases) will leave `uuid` as `NULL`, triggering a database integrity violation or — worse — silently omitting the UUID if the column is somehow nullable. Even if the column has a `NOT NULL` constraint, the resulting DB exception will be confusing rather than self-documenting. The UUID must be owned by the model.

**Fix:**
```php
// app/Models/Payment.php

use Illuminate\Support\Str;

protected static function boot(): void
{
    parent::boot();

    static::creating(function (Payment $payment): void {
        if (empty($payment->uuid)) {
            $payment->uuid = (string) Str::uuid();
        }
    });
}
```

Also add the `uuid` cast so Eloquent treats it as a string consistently:
```php
protected function casts(): array
{
    return [
        'uuid'       => 'string',
        'amount'     => 'integer',
        'expires_at' => 'datetime',
        'paid_at'    => 'datetime',
    ];
}
```

### CR-02: PaymentFactory Creates Mismatched Brand / StripeAccount

**File:** `database/factories/PaymentFactory.php:17-18`

**Issue:** `brand_id` and `stripe_account_id` are each resolved via independent factory calls, which creates two separate `Brand` records and one `StripeAccount` that belongs to the second brand — not to the brand referenced by `brand_id`. Every factory-created `Payment` therefore violates the business invariant that a payment's `stripe_account_id` must belong to the same brand as `brand_id`. This will cause silent data integrity failures in any future test that traverses `$payment->stripeAccount->brand`.

```php
// Current — wrong: two different Brand records
'brand_id'          => Brand::factory(),         // creates Brand A
'stripe_account_id' => StripeAccount::factory(), // creates StripeAccount → Brand B
```

**Fix:** Create the brand once and pass it to both:
```php
public function definition(): array
{
    $brand = Brand::factory()->create();

    return [
        'uuid'                     => Str::uuid(),
        'brand_id'                 => $brand->id,
        'stripe_account_id'        => StripeAccount::factory()->create(['brand_id' => $brand->id])->id,
        'user_id'                  => User::factory(),
        'amount'                   => $this->faker->numberBetween(500, 100000),
        'currency'                 => $this->faker->randomElement(['usd', 'gbp']),
        'description'              => $this->faker->sentence(),
        'status'                   => 'pending',
        'client_email'             => $this->faker->safeEmail(),
        'stripe_payment_intent_id' => null,
        'expires_at'               => null,
        'paid_at'                  => null,
    ];
}
```

### CR-03: Encrypted Credential Fields Exposed via Mass Assignment

**File:** `app/Models/StripeAccount.php:14-17`

**Issue:** `secret_key` and `webhook_secret` are listed in `$fillable`. While the `encrypted` cast protects them at rest, including them in `$fillable` means any future controller that passes `$request->all()` or `$request->validated()` to `StripeAccount::create()` / `$account->fill()` will accept attacker-controlled Stripe credentials — replacing a brand's live keys with attacker keys and redirecting payments. This is a mass-assignment security vulnerability waiting to be triggered in Phase 3.

**Fix:** Remove the credential fields from `$fillable` and assign them explicitly in service/controller code:
```php
protected $fillable = [
    'brand_id',
    'account_name',
    'publishable_key',
    'is_active',
    // secret_key and webhook_secret are NOT mass-assignable
    // Assign explicitly: $account->secret_key = $validated['secret_key'];
];
```

## Warnings

### WR-01: Payment Model Uses Default Integer Route Key — Exposes Sequential IDs

**File:** `app/Models/Payment.php:1-42`

**Issue:** Laravel's default `getRouteKeyName()` returns `'id'`. Payment links (Phase 4) will be routed by integer ID, enabling trivial enumeration of all payments. The `uuid` column exists precisely for public-facing identification but is never nominated as the route key.

**Fix:**
```php
public function getRouteKeyName(): string
{
    return 'uuid';
}
```

### WR-02: EncryptionRoundTripTest Missing Null-Check on webhook_secret Raw Value

**File:** `tests/Feature/EncryptionRoundTripTest.php:40`

**Issue:** The `webhook_secret` encryption test (line 29–43) checks `not->toBe($plainSecret)` but omits the `->not->toBeNull()` assertion that the `secret_key` test includes at line 21. If encryption silently fails and stores `NULL`, the test still passes because `NULL !== 'whsec_placeholder_for_dev_only'`.

**Fix:**
```php
$rawRow = DB::table('stripe_accounts')->where('id', $account->id)->value('webhook_secret');
expect($rawRow)->not->toBe($plainSecret);
expect($rawRow)->not->toBeNull(); // add this line
```

### WR-03: PaymentAmountIntegrityTest Missing ->toBeInt() Assertion on Minimum Amount

**File:** `tests/Feature/PaymentAmountIntegrityTest.php:25-29`

**Issue:** The "stores minimum valid amount" test (line 25) asserts `toBe(50)` but does not assert `toBeInt()`, unlike the first two tests. If the cast were removed or broken and the value returned as the string `"50"`, `toBe(50)` would fail (strict type check in Pest/PHPUnit), but the intent is unclear and inconsistent with the pattern established on lines 13 and 21.

**Fix:**
```php
it('stores minimum valid amount without drift', function () {
    $payment = Payment::factory()->create(['amount' => 50]);
    $fresh   = Payment::find($payment->id);

    expect($fresh->amount)->toBe(50);
    expect($fresh->amount)->toBeInt(); // add for consistency
});
```

### WR-04: Brand Primary Color Injected Into CSS Custom Property Without Sanitization

**File:** `resources/css/brand-theme.css:9-13`

**Issue:** The comment on line 9 documents that Phase 5 will apply brand colors via inline style: `:style="\`--brand-primary: ${brand.primary_color}; ...\`"`. The `primary_color` and `secondary_color` columns are stored as free-text `string(7)` in the database (migration line 19). If a malicious or malformed value reaches this inline style — e.g., `red); --some-other-var: url(javascript:` — it would break CSS parsing or, in older browser contexts, enable CSS injection. The 7-character column length provides some protection, but there is no model-level or migration-level validation enforcing the `#RRGGBB` format.

**Fix:** Add a regex validation rule when brand colors are saved (in the Phase 3 form request / controller), and add a computed accessor or helper that sanitizes the value before rendering:
```php
// In a FormRequest for brand creation/update:
'primary_color'   => ['required', 'regex:/^#[0-9A-Fa-f]{6}$/'],
'secondary_color' => ['required', 'regex:/^#[0-9A-Fa-f]{6}$/'],
```

Consider also enforcing the format at the migration level:
```php
$table->string('primary_color', 7)->default('#3B82F6');
// Add a CHECK constraint in a raw statement if your DB supports it
```

use Illuminate\Contracts\Auth\MustVerifyEmail;

### IN-02: stripe_payment_intent_id Is Mass-Assignable

**File:** `app/Models/Payment.php:13-17`

**Issue:** `stripe_payment_intent_id` is in `$fillable`. This field should only ever be written by server-side webhook handling (Phase 6), never from user input. Listing it as mass-assignable increases the surface area for accidental assignment from a request payload.

**Fix:** Remove it from `$fillable` and assign it only in the webhook handler:
```php
protected $fillable = [
    'uuid', 'brand_id', 'stripe_account_id', 'user_id',
    'amount', 'currency', 'description', 'status',
    'client_email', 'expires_at', 'paid_at',
    // stripe_payment_intent_id assigned explicitly by webhook handler only
];
```

### IN-03: Seeder Uses Weak Hardcoded Admin Password

**File:** `database/seeders/DatabaseSeeder.php:45`

**Issue:** The admin user is seeded with `Hash::make('password')`. If this seeder is ever run against a staging or production environment (e.g., during a demo deploy), the admin account is immediately compromised. The password is not even noted as requiring change.

**Fix:** Either drive it from an environment variable or add a conspicuous comment:
```php
'password' => Hash::make(env('ADMIN_SEED_PASSWORD', 'change-me-before-deploy')),
```
Or throw if the env var is absent in non-local environments:
```php
if (app()->isProduction()) {
    throw new \RuntimeException('DatabaseSeeder must not run in production.');
}
```
