# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Phase 1 scope note:** Phase 1 has no real application UI. The design contract here establishes the design system foundation — tokens, the shadcn-vue initialization, and the five seed components — that every downstream phase (2–7) will consume. The only runtime UI deliverable is the Inertia root page confirming Tailwind renders. Breeze auth pages ship with the Laravel starter kit and are intentionally left unstyled in this phase.

```bash
npx shadcn-vue@latest init
```

Confirm `components.json` exists after init before proceeding to component seeding.

## Multi-Brand Theming Architecture

This is a design system decision that must be established in Phase 1, consumed by Phase 5.

### CSS Variable System

Create `resources/css/brand-theme.css` with this structure:

```css
/* Default (admin panel) — neutral, no brand coloring */
:root {
  --color-brand-primary: hsl(221.2 83.2% 53.3%);   /* shadcn-vue blue default */
  --color-brand-secondary: hsl(210 40% 96.1%);       /* shadcn-vue muted default */
}

/* Per-brand overrides — applied via data-brand attribute on layout root */
[data-brand] {
  --color-brand-primary: var(--brand-primary, hsl(221.2 83.2% 53.3%));
  --color-brand-secondary: var(--brand-secondary, hsl(210 40% 96.1%));
}
```

### Runtime brand injection (Phase 5)

When the client payment page loads, the Vue layout component receives brand colors as Inertia page props and injects them as inline CSS variables on the `<html>` or top-level layout `<div>`:

```vue
<!-- resources/js/layouts/BrandLayout.vue (Phase 5) -->
<div :data-brand="brand.slug"
     :style="`--brand-primary: ${brand.primary_color}; --brand-secondary: ${brand.secondary_color};`">
```

This is established as the pattern in Phase 1 so Phases 3–5 build toward it consistently.

### Admin panel

The admin panel uses NO `data-brand` attribute. It renders in shadcn-vue's default neutral palette. Brand colors only appear on the client-facing payment pages.

## Spacing Scale

Standard 8-point scale. Declared values are multiples of 4.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, tight list item gaps |
| sm | 8px | Compact element spacing, input padding supplement |
| md | 16px | Default element spacing, form field gaps |
| lg | 24px | Section padding, card body padding |
| xl | 32px | Layout gaps between major sections |
| 2xl | 48px | Major section breaks, page-level vertical rhythm |
| 3xl | 64px | Page-level top/bottom padding |

Exceptions: Touch targets on mobile must meet 44px minimum (Tailwind `min-h-[44px]`). Apply to Button and interactive Card components starting Phase 5.

## Typography

Four sizes, two weights. Applied via Tailwind utility classes using shadcn-vue's CSS variable font system.

| Role | Size | Weight | Line Height | Tailwind Class | Usage |
|------|------|--------|-------------|----------------|-------|
| Body | 14px | 400 (regular) | 1.5 | `text-sm font-normal leading-normal` | Form helper text, table cells, descriptions |
| Label | 14px | 500 (medium) | 1.4 | `text-sm font-medium leading-snug` | Form field labels, badge text |
| Heading | 20px | 600 (semibold) | 1.25 | `text-xl font-semibold leading-tight` | Page section headings, card titles |
| Display | 28px | 600 (semibold) | 1.2 | `text-2xl font-semibold leading-tight` | Page-level headings (Phase 2+ login page, dashboard header) |

Weights in use: regular (400) and semibold (600). Medium (500) is used for labels only as a midpoint — do not introduce additional weights.

## Color

### Admin Panel Palette (neutral — no brand colors)

shadcn-vue New York style ships a complete neutral palette via CSS variables. No additional custom colors are declared at this layer.

| Role | shadcn-vue Variable | Approximate Hex | Usage |
|------|---------------------|-----------------|-------|
| Dominant (60%) | `--background` | `#ffffff` (light) / `#09090b` (dark) | Page background, surface background |
| Secondary (30%) | `--card`, `--muted` | `#f4f4f5` (light) | Cards, sidebar, nav background, input background |
| Accent (10%) | `--primary` | `#18181b` (light) / `#fafafa` (dark) | Primary buttons, active nav item, focus rings |
| Destructive | `--destructive` | `#ef4444` | Destructive action buttons only |

Accent (`--primary`) is reserved for: primary Button component, active navigation state. It must NOT be applied to decorative elements, card borders, or informational text.

### Client Payment Page Palette (branded — Phase 5)

| Role | CSS Variable | Value | Usage |
|------|-------------|-------|-------|
| Brand primary | `--color-brand-primary` | Set from `brand.primary_color` at runtime | CTA button color, focus ring on Stripe Elements, link color |
| Brand secondary | `--color-brand-secondary` | Set from `brand.secondary_color` at runtime | Section background, header/footer background |
| Page background | `--background` (shadcn-vue default) | `#ffffff` | Payment page canvas |
| Destructive | `--destructive` (shadcn-vue default) | `#ef4444` | Error state on payment failure page |

Brand colors are stored in the `brands` table as hex strings (`primary_color`, `secondary_color`). Phase 3 adds the color picker to the brand creation form. Phase 5 consumes them as CSS variables.

### Status Badge Color Map (Phase 4–7)

| Status | Badge Variant | Meaning |
|--------|--------------|---------|
| `pending` | `secondary` | Payment link created, not yet paid |
| `completed` | `default` (with green override via `--primary`) | Payment succeeded via webhook |
| `failed` | `destructive` | Payment failed via webhook |
| `cancelled` | `outline` | Manually cancelled |

Implement badge status coloring with Tailwind variant overrides, not hard-coded hex values. Phase 4 builds this.

## Copywriting Contract

Phase 1 has no user-facing copywriting. The entries below establish the copy conventions for downstream phases.

| Element | Copy | Phase |
|---------|------|-------|
| Primary admin CTA | "Create Payment" | Phase 4 |
| Primary client CTA (Stripe Elements submit) | "Pay [amount]" | Phase 5 |
| Empty payment list | "No payments yet. Create your first payment to get started." | Phase 4 |
| Payment link copy action | "Copy Link" (button label); toast: "Link copied to clipboard" | Phase 4 |
| Error state — payment not found | "This payment link is not valid or has already been completed." | Phase 5 |
| Error state — payment failed | "Your payment could not be processed. Please check your card details and try again." | Phase 5 |
| Destructive: deactivate Stripe account | Label: "Deactivate Account" — confirmation: "This account will no longer be available for new payments. Existing payment links remain active until paid." | Phase 3 |
| Destructive: delete brand | Not supported in v1 — no delete action, no confirmation needed | N/A |

### Copywriting Conventions

- Use sentence case for all labels and headings. Example: "Create payment" not "Create Payment" — exception: proper nouns (Stripe, PayHub, brand names).
- Errors must state the problem AND the next step. Never "An error occurred."
- Amount display: always show currency symbol + amount with 2 decimal places. Example: "$125.00" or "£49.99". Never "125" or "USD 125".
- Amounts in USD show "$", amounts in GBP show "£". No ISO code suffix in user-facing text.
