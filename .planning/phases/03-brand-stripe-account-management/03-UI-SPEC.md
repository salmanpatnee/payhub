## Spacing Scale

Inherited from Phase 2 contract. No changes.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, color swatch inline padding, badge padding |
| sm | 8px | Field group internal gaps, row gaps within grid |
| md | 16px | Default form field spacing, card padding, page padding |
| lg | 24px | Section spacing between form groups, page content padding |
| xl | 32px | Major vertical spacing between page sections |
| 2xl | 48px | Not used in Phase 3 |
| 3xl | 64px | Not used in Phase 3 |

Exceptions:
- Touch targets on interactive icon buttons: minimum 44px (achieved via shadcn Button `size="icon"` — already established)
- Color picker native `<input type="color">`: 32px × 32px visual target, wrapped in a flex row alongside a text Input for hex value
- Brand preview card: no max-width constraint — fluid within the form column width

## Typography

Inherited from Phase 2 contract. All type set in Instrument Sans via `--font-sans`.

NOTE: The implemented codebase uses `font-semibold` (600) where the Phase 2 UI-SPEC documented `font-medium` (500). Match the actual code — use `font-semibold` for headings and table headers.

| Role | Size | Weight | Line Height | Tailwind Class | Usage |
|------|------|--------|-------------|----------------|-------|
| Body | 14px | 400 (regular) | 1.5 | `text-sm` | Table cell content, form helper text, preview card brand name body copy |
| Label | 14px | 600 (semibold) | 1.4 | `text-sm font-semibold` | Form field labels, table column headers |
| Heading | 20px | 600 (semibold) | 1.2 | `text-xl font-semibold` | Page titles ("Brands", "Stripe Accounts") — matches actual Index.vue `text-xl font-semibold` |
| Display | 28px | 600 (semibold) | 1.2 | `text-2xl font-semibold` | Not used in Phase 3 — reserved for Phase 7 dashboard |

Active weights in Phase 3: **400 (regular)** and **600 (semibold)**. No weight 500.

Muted text: `text-muted-foreground` — used for table column headers, secondary info, placeholder copy, key masked text.

## Color

All colors resolved from CSS custom properties in `app.css`. Inherited from Phase 2. No new tokens.

| Role | CSS Token | Light Mode Value | Usage |
|------|-----------|-----------------|-------|
| Dominant (60%) | `--background` | `hsl(0 0% 100%)` — white | Page background, main content area |
| Secondary (30%) | `--card` / `--sidebar-background` | `hsl(0 0% 100%)` / `hsl(0 0% 98%)` | Form card surface, sidebar surface |
| Accent (10%) | `--primary` | `hsl(0 0% 9%)` — near-black | Primary CTA buttons, active sidebar nav item |
| Destructive | `--destructive` | `hsl(0 84.2% 60.2%)` — red | Deactivate Stripe account button + confirmation |

Accent (`--primary`) reserved for:
1. "Add brand" CTA button on the brand list page
2. "Create brand" submit button on the brand create form
3. "Save changes" submit button on the brand edit form
4. "Add Stripe account" CTA button on the Stripe accounts list page
5. "Save account" submit button on the Stripe account create/edit form
6. Active sidebar navigation item highlight (already wired, no change)

Destructive state reserved exclusively for:
- "Deactivate" button on a Stripe account row (not hard delete — deactivation toggle)
- Deactivation confirmation dialog confirm button

**Brand color swatches in the list:** Brand `primary_color` and `secondary_color` hex values are displayed as inline `<span>` elements with `background-color` set via `:style` binding. These are decorative display elements — they use `background-color` directly, NOT Tailwind color classes (Tailwind 4 purges dynamic class names at build time).

**Live preview card colors:** The brand preview card uses `:style` bindings driven by reactive hex values from the form. No Tailwind dynamic color classes. Pattern:
```vue
:style="{ backgroundColor: form.primary_color }"
```

**Stripe API error color:** Use Alert component with `variant="destructive"` for Stripe API validation errors. This uses `--destructive` — the same semantic token as all other error states.

**Test key error color:** Use Alert component with `variant="destructive"` for test-mode key blocking error. Same token, same component.

Badge colors for Stripe account status:
- Active account (`is_active = true`): `variant="secondary"` — neutral, not alarming
- Inactive account (`is_active = false`): `variant="outline"` — visually subdued

## Interaction States

### Form Submission (all forms)

| State | Visual |
|-------|--------|
| Idle | Submit button enabled, normal label |
| Processing | Button disabled, Spinner prepended to label (`:disabled="form.processing"`) |

| Validation error (standard) | Stay on page; InputError shown below each invalid field; no toast |

| Test key blocked (BRAND-04) | Stay on page; Alert `variant="destructive"` shown at top of form with specific message |
| Server error (500) | Inertia default error page — no custom handling |

### Color Input Sync

Native `<input type="color">` and shadcn `Input` text field share the same `v-model` binding. Both update the same reactive ref:
- User picks via color wheel → hex text field updates to the chosen hex value
- User types hex value in text field → color wheel updates on valid 6-char hex
- Invalid partial input (e.g. `#FF`) does NOT update the preview card — preview only updates when `form.primary_color` matches `/^#[0-9a-fA-F]{6}$/`

No debounce needed — Tailwind purge is not a concern here since all brand colors use `:style` bindings.

### Live Preview Card Reactivity

Preview card updates on every keystroke in color fields and every logo file selection. No explicit "preview" button. Preview shows fallback values when fields are empty:
- Brand name empty → show "Brand name" placeholder text
- No logo selected → show brand initial letter in a white/20 circle
- Primary color invalid → fall back to `#000000` for preview rendering
- Secondary color invalid → fall back to `#cccccc` for preview rendering

### Deactivation Flow

1. User clicks PowerOff icon button in the Stripe accounts list
2. Dialog opens: "Deactivate account?" with account name in description
3. User clicks "Keep active" → dialog closes, no action
4. User clicks "Deactivate" → `deactivateForm.patch()` submitted, button disabled during processing
5. On success: dialog closes, page reloads via Inertia redirect, Sonner toast: "Account deactivated."

### Logo File Selection

1. User selects file via `<input type="file">`
2. Vue `@change` handler creates `URL.createObjectURL(file)` and sets `logoPreviewUrl` ref
3. Preview card updates immediately to show the selected image
4. On form submit, `useForm()` handles the file as multipart
5. On validation failure (wrong type, too large): InputError shown below the file input

### Row Hover State

All table rows: `hover:bg-muted/50 transition-colors` — inherits from Phase 2 pattern. Applied uniformly.

## Accessibility Notes

- All form fields have explicit `<Label for="...">` with matching `id` on the input (established Phase 2 pattern — maintain for all brand and Stripe account forms)
- Native `<input type="color">` must have `aria-label` ("Pick primary color" / "Pick secondary color") since it has no visible text label of its own — the Label above labels the group, not the native input
- Color swatch `<span>` elements in the brand list are decorative — add `aria-hidden="true"` and a `title` attribute for hover tooltip with the hex value
- Row action icon buttons must have `aria-label="Edit {name}"`, `aria-label="View Stripe accounts for {name}"`, `aria-label="Deactivate {name}"` — same pattern as Phase 2
- Dialog component from shadcn-vue handles focus trap and `aria-modal` automatically — no additional implementation needed
- The secret key field uses `type="password"` — this suppresses browser autofill for credentials and visually masks the value during typing
- Live preview card is `aria-hidden="true"` — it is a decorative preview, not interactive or informational for screen readers
- Logo thumbnail `<img>` elements use `alt=""` — the brand name in adjacent text provides the semantic context
